# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What this is

A .NET 10 console tool (`ColumnstoreAnalyzer`) that connects to a live SQL Server database and ranks its large tables by how good a candidate each is for a columnstore index. Unlike DMV-only advisors, it **reads the actual column data** to measure repetitiveness, because columnstore compression (dictionary + run-length encoding) depends on duplicate values. See `README.md` for the full domain rationale and the scoring philosophy.

## Build & run

```bash
dotnet build -c Release
dotnet run -c Release -- --server SQLPROD01 --database SalesDW              # Windows integrated auth
dotnet run -c Release -- --server SQLPROD01 --database SalesDW --user analyst  # SQL auth, prompts for masked password
dotnet publish -c Release -r win-x64 --self-contained                       # standalone exe
dotnet test ColumnstoreAnalyzer.Tests -c Release                            # unit tests (no DB needed)
```

There's no linter config or CI in this repo. `ColumnstoreAnalyzer.Tests/` (xUnit) covers the pure-logic pieces that don't need a live SQL Server — `BlitzAdapter`, `Scoring`, `PermissionGrouping`, `CsvUtil`, `FindingsUtil` — via `InternalsVisibleTo` from the main project; it's excluded from the main project's own compile via an explicit `<Compile Remove>` since it's nested in the same directory tree and would otherwise get swept up by the default glob. Everything that actually touches a database can't be unit-tested here (no test SQL Server exists) — that's what `--self-test` is for (see below): a read-only mode meant to be run manually against a real instance. `--help` prints full CLI usage. Target DB permissions required: `VIEW SERVER STATE` + read on the database (the opt-in health-check stage widens this — see below).

## Architecture

The program is a **single linear pipeline** with no DI, no async, and one class per stage — with one exception: `--permissions-report` is a fully standalone mode that runs *instead of* this pipeline (see "Permissions inventory" below), not another stage bolted onto it. For everything else, `Program.Run` (`Program.cs`) is the orchestrator and reads top-to-bottom as the whole flow:

1. `Analyzer.DiscoverTables` — finds large user tables (size/rowcount thresholds) with workload + columnstore/replication flags in one query.
2. If `--health-check` was passed: `HealthCheckRunner.Run` (see "Health check stage" below) — runs right after server-vitals, before table discovery.
3. Per table, in order: `LoadOperationalStats`, `LoadIndexes`, `LoadColumns`, then `AnalyzeColumnCardinality` (skipped if the table already has a columnstore index).
4. `MatchPlanCacheQueries` — one pass over the whole plan cache, statements matched to tables **client-side by table-name substring**.
5. `MatchQueryStoreQueries` — same table-name-substring matching, but against Query Store's persisted, restart-surviving query history for the connected database (supplements, never replaces, the plan-cache scrape above — entirely additive, and a no-op if Query Store is off). Both merge into the same `TableInfo.ReferencingQueries` list; `CachedQuery.Source` ("PlanCache"/"QueryStore") tracks which supplied a given row.
6. `Scoring.Score` per table → 0–100 candidacy score.
7. `ReportWriter.WriteAll` → CSVs + `full_report.json` (+ `7_health_check_findings.csv`/`health_check.json` if health-check ran); `Narrative.Build` → `5_analysis.md`; optional `LlmClient.GenerateNarrative` → `6_llm_analysis.md`; `HtmlReportWriter.Write` → `report.html` (self-contained dashboard, on by default, `--no-html-report` to skip).

Key module responsibilities:
- **`Sql.cs`** — every columnstore-analysis T-SQL query as a `const string`. All DB queries live here; `Analyzer.cs` only executes them and maps rows to models. Change SQL here, not inline.
- **`Analyzer.cs`** — the columnstore-analysis DB-touching class. Opens a fresh `SqlConnection` per call (`Open()` → `ConnectionFactory.Open(opt)`). `AnalyzeColumnCardinality` is the heart: batches ~20 columns per query (`COUNT_BIG(DISTINCT)`, null count, avg `DATALENGTH`) under `READ UNCOMMITTED` + `MAXDOP 2`, using `TABLESAMPLE SYSTEM` when a table exceeds `SampleTargetRows * 2`. `MatchQueryStoreQueries` checks `sys.database_query_store_options` first and bails with a caveat string (surfaced as a console `NOTE:`) if Query Store is off or stuck `READ_ONLY` (the well-known "hit its storage cap and silently stopped capturing data" gotcha, via the `readonly_reason` bitmask — that bitmask has 7 documented bits, not just the storage one; re-check Microsoft's docs before changing which count as "bad" vs. benign like single-user mode or secondary replica) — otherwise fetches Query Store's top queries in the same aggregated-totals shape as `MatchPlanCacheQueries` and merges them in.
- **`ConnectionFactory.cs`** — the one place the `SqlConnectionStringBuilder`/auth logic lives, shared by all three DB-touching classes (`Analyzer`, `HealthCheckAnalyzer`, `PermissionsAnalyzer`) so it can't drift between them.
- **`Models.cs`** — all data types. `TableInfo` / `ColumnStat` hold raw DMV values as `init`/`set` properties and expose **derived metrics as computed getters** (`ScanPct`, `WritePct`, `DistinctRatio`, `DuplicationFactor`, `DictionaryPressure`, `Verdict`, `NcIndexBloatRatio`, `PotentialFullRowgroups`). Put derived logic in these getters, not in the analyzer or scorer. Also holds the health-check types (`HealthCheckFinding`, `ComponentStatus`, `MaintenanceJobStatus`, `DynamicResultSet`, `HealthCheckResult`, `InstallAction`) and the permissions-report types (`ServerPrincipalInfo`, `DatabaseUserInfo`, `ObjectPermissionInfo`, `PermissionsReportResult` — the last of these reuses `HealthCheckFinding`/`HealthCheckSeverity` for its computed risk findings rather than inventing a parallel type) and their `AnalyzerOptions` flags.
- **`Scoring.cs`** — pure function `Score(TableInfo)` mutating `CandidacyScore` + `AssessmentNotes`. The weighting model (data-shape up to 40 pts, size 15, scan 20, NC-index bloat 15, contention 10, minus write/update/dictionary/LOB penalties) is the product's opinion — keep it in sync with the rationale in the class doc-comment and README. Scoped to columnstore candidacy only — health-check findings live in `HealthCheckFinding.Severity`, not here.
- **`Narrative.cs`** — rule-based (no AI) DBA-style prose report for columnstore candidacy; the LLM output is strictly additive grounding, never required. Not extended for health-check findings (those get their own dashboard in `HtmlReportWriter.cs`) — avoid scope creep into a second markdown narrative generator.
- **`LlmClient.cs`** — talks to any OpenAI-compatible endpoint (Ollama/LM Studio). Purely optional; failures are caught and non-fatal. This is the reference pattern the health-check stage's third-party integrations (FRK/Ola Hallengren) follow: opt-in, off by default, non-fatal.
- **`ReportWriter.cs`** — the numbered CSVs, `full_report.json`, and (when health-check ran) `7_health_check_findings.csv` + a separate `health_check.json` (kept separate so `full_report.json`'s existing schema stays untouched for anyone already parsing it).
- **`HtmlReportWriter.cs`** — self-contained `report.html` dashboard (inline CSS/JS only, no CDN — must open with zero network access). An *additional* artifact, not a replacement for the CSVs/markdown. Hand-rolled `StringBuilder` HTML, same style as `Narrative.cs`; uses `System.Net.WebUtility.HtmlEncode` for anything embedded (query text, finding details) since that content isn't guaranteed HTML-safe.
- **`PasswordInput.cs`** — masked interactive password prompt, stdin-redirect aware. `HealthCheckRunner`'s install-flow confirmation prompt follows the same stdin-redirect-aware pattern (abort rather than silently proceed when non-interactive).
- **`CsvUtil.cs`**, **`PermissionGrouping.cs`**, **`FindingsUtil.cs`** — small pure-logic helpers extracted specifically so `ColumnstoreAnalyzer.Tests` can exercise them without a database: shared CSV field-escaping, the grant-grouping/target-labeling logic behind the permissions report's dedup, and the health-check findings dedup/database-filter pass. If you add a new piece of DB-independent logic, prefer a similarly small `internal static` class here over a private method buried in a runner/writer — that's the difference between testable and not.

### Health check stage (opt-in, `--health-check`, off by default)

A second, parallel pipeline for broader SQL Server best-practices checks (Brent Ozar's First Responder Kit, Ola Hallengren's Maintenance Solution, sp_WhoIsActive, plus native checks and a "tribal knowledge" inventory), designed so a departing DBA/engineer's institutional knowledge survives them. Entirely additive — default behavior, output, and permissions are unchanged unless `--health-check` is passed.

- **`SqlHealthCheck.cs`** / **`HealthCheckAnalyzer.cs`** — the health-check equivalent of `Sql.cs`/`Analyzer.cs`. `HealthCheckAnalyzer` is the second of the tool's three DB-touching classes; same shape (`Open()`, one method per check, no internal catching — that's the runner's job). Includes `DynamicResultSet`/`ReadAllDynamic` for capturing EXEC'd stored procs whose result schema isn't known in advance (FRK procs can also return >1 result set — `ReadAllDynamic` loops `NextResult()`).
- **`BlitzAdapter.cs`** — pure mapper, no DB access (like `Scoring.cs`), `DynamicResultSet → List<HealthCheckFinding>`. Defensive by design: unrecognized/renamed columns degrade to a raw-row `Info` finding rather than throwing, since FRK's exact schema can change between releases. The `Priority`→severity breakpoints are a best-effort reading of Ozar's convention, not verified against a live run — sanity-check before trusting them blindly.
- **`HealthCheckRunner.cs`** — orchestrates the whole stage with the same non-fatal per-step console pattern as `Program.Run`. `sp_WhoIsActive` is **detect-only, never executed** — it's a live-snapshot tool, meaningless in a static report. `sp_BlitzIndex`/`sp_BlitzCache`/`sp_BlitzLock` are scoped to `--database` via `@DatabaseName` (verified against the pinned FRK source — `sp_Blitz` and `sp_BlitzBackups` have no such parameter upstream and stay instance-wide by design); `sp_BlitzFirst` has no `@DatabaseName` either but takes `@FilterPlansByDatabase` for its embedded plan-cache portion. `sp_BlitzFirst` always runs its instant snapshot (`@SinceStartup=1`) and, by default, *also* captures a supplementary 30-second live `@Seconds=` sample (`--blitzfirst-seconds N` to change the duration, `0` to skip it) — this is the one place in the health-check stage that intentionally burns real wall-clock time by default; keep it clearly labeled in the console progress line.
- **`InstallSources.cs`** — pinned metadata (exact release tag/URL + verified SHA-256) for the opt-in auto-install feature (`--health-check --install-missing-tools`). **Never "fetch latest"** — this is the one part of the tool that can write to the target instance (`CREATE PROCEDURE`, SQL Agent jobs), so it only runs a script whose content has been reviewed once and pinned, gated behind interactive confirmation, and aborts safely under piped/non-interactive input. Brent Ozar's own 2026 FRK release notes explicitly warn against auto-fetching-and-running code from the internet against SQL Server for exactly this reason — re-verify and update the checksum here deliberately when intentionally upgrading, never loosen this without re-adding equivalent rigor.
- **Availability Group / replication checks** (also in `HealthCheckAnalyzer.cs`/`SqlHealthCheck.cs`) — real sync-state/lag health via `sys.dm_hadr_availability_replica_states`/`sys.dm_hadr_database_replica_states` (needs `VIEW SERVER PERFORMANCE STATE` on SQL 2022+, `VIEW SERVER STATE` on 2019 and earlier — both DMVs return zero rows, not an error, on an instance with no AG at all, so no special "is this instance in an AG" branch is needed). This replaced the old shallow `ag_replica_count` in `RunTopologyRollup`, which only counted replicas and said nothing about their health. Legacy transactional/snapshot/merge replication gets a deliberately lighter treatment — recent error surfacing from the distributor's `MSrepl_errors` only, not a full latency deep-dive; don't expand this without a real reason, it was scoped down on purpose.

### Permissions inventory (standalone, `--permissions-report`)

A logins/permissions security checkup across **every** database on the instance, for sysadmins doing access-inventory/audit work. Unlike the health-check stage, this is not additive to the columnstore pipeline — `Program.Run` branches to `RunPermissionsReport` and returns immediately when the flag is set, so `--database` is not required (and is ignored if given).

- **`SqlPermissions.cs`** / **`PermissionsAnalyzer.cs`** — the third and last DB-touching class. Key implementation fact: `sys.database_principals`/`sys.database_role_members`/`sys.database_permissions`/`sys.objects`/`sys.schemas` are per-database **catalog views**, which (unlike most `sys.dm_*` dynamic management views) resolve correctly when queried via a 3-part name (`[OtherDB].sys.database_principals`) from any connection — so this needs only one connection and one dynamic query per database, never per-database connection switching. Orphaned-user detection cross-references each database user's `sid` against `sys.server_principals.sid`, and correctly excludes `authentication_type_desc <> 'INSTANCE'` users (contained-DB users are by design login-less, not orphaned).
- **`PermissionsReportRunner.cs`** — orchestrates: server principals/roles once, then loops every online user database, catching per-database failures as warnings (a login without `VIEW DEFINITION` on some database is expected, not fatal — see the permission-model caveat in `--help`). Computes risk findings at the end (sysadmin inventory, disabled logins that still have live database access, orphaned users, risky object grants — `CONTROL`/`ALTER`/`TAKE OWNERSHIP` or anything granted to `public`) as `HealthCheckFinding`s, reusing that type rather than inventing a parallel one.
- **`PermissionsReportWriter.cs`** / **`PermissionsNarrative.cs`** / **`PermissionsHtmlWriter.cs`** — four output formats total: CSV (3 files) + JSON from the first, an exec-summary markdown from the second (counts/findings/per-database summary table, deliberately NOT a full row dump — that's what the CSV/JSON are for), and a self-contained `permissions_report.html` from the third (own file, separate from `report.html`, since this mode never runs alongside the columnstore pipeline — its CSS block is a deliberate copy of `HtmlReportWriter.cs`'s, not a shared one, since the two dashboards have no other coupling).

### Self-test (standalone, `--self-test`)

A strictly read-only diagnostic mode, meant to be run manually by someone with real access to a live instance and the resulting file handed back for review — this tool has no live database access of its own during development, so this is the only way bugs in the DB-touching code get caught before a user hits them. Also runs instead of every other mode; `--database` not required.

- **`SelfTestRunner.cs`** — exercises every code path already built (`Analyzer`, `HealthCheckAnalyzer`, `PermissionsAnalyzer`) rather than duplicating any SQL: a small sample of columnstore tables (capped at `MaxTablesToExercise`, currently 3, for speed on a real prod instance) including `MatchQueryStoreQueries`, every health-check native check including AG health, FRK detection + **instant-only** EXEC of whatever's installed (never the 30s live sample, regardless of `--blitzfirst-seconds` — self-test always stays fast and predictable), Ola detection, and the full permissions inventory. Every step is wrapped by a `Step`/`StepValue` helper that times it, catches any exception, and records pass/fail — a failing step never stops the others.
- **Never touches the install-flow.** `RunInstallFlow` is simply never called from this code path, and `Program.ParseArgs` additionally refuses to run if `--self-test` and `--install-missing-tools` are both passed — belt-and-suspenders, since the whole point of this mode is being safe to point at production.
- **`SelfTestWriter.cs`** — writes `self_test_results.json` (precise) and `self_test_results.md` (human-skimmable) and prints a pass/fail summary to the console. Step summaries deliberately capture row/column **counts and shape**, never actual data values, since the point is to be safe to run (and share) against a real production instance — if you add a new self-test step, keep following that rule.

## Conventions

- **Identifier escaping**: any table/column/database name interpolated into SQL goes through `Analyzer.Escape` (`internal static`, `[...]` with `]` doubled) — shared by all three DB-touching classes. Never interpolate a raw identifier into a query string. (DBCC DBINFO's database-name argument is a string *literal*, not a bracketed identifier — that one uses single-quote doubling instead, see `HealthCheckAnalyzer.RunCheckdbAge`.)
- **Non-fatal by design**: per-table analysis failures, plan-cache scrape, the LLM step, every health-check step, every per-database step in the permissions report, and every step in `--self-test` all catch and continue — one bad check (or one database the login can't see into) must not abort the run. Preserve this.
- **DMV caveats are load-bearing**: sampled distinct counts are within-sample only; usage stats reset on instance restart (tool warns when uptime < 7 days); plan-cache matching is substring-based. Don't "fix" these into implying more precision than they have.
- File-scoped namespace `ColumnstoreAnalyzer`, nullable + implicit usings enabled, collection expressions (`[]`), switch expressions. Match the existing terse style.
